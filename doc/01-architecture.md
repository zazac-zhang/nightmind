# 整体架构设计

## 1. 架构概览

NightMind 采用分层架构，从上到下分为：API 层、核心层、服务层、数据层。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                              NightMind Architecture                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌─────────────┐      ┌──────────────────────────────────────────────────┐ │
│  │   Clients   │      │              API Layer (Axum)                     │ │
│  │ (Web/Mobile)│◄────►│  WebSocket / HTTP Router / Middleware            │ │
│  └─────────────┘      └──────────────────────────────────────────────────┘ │
│                              │                                               │
│                              ▼                                               │
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │                        Core Layer (Rig + Agents)                      │ │
│  │  SessionManager / TopicStack / StateSnapshotter / NightMindAgent     │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                              │                                               │
│                              ▼                                               │
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │                         Service Layer                                 │  │
│  │  AudioService / STTService / TTSService / Integration Services       │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                              │                                               │
│                              ▼                                               │
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │                      Data & Storage Layer                             │  │
│  │  PostgreSQL / Redis / Vector Store (Qdrant) / File Storage           │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 2. 技术栈选型

### 2.1 核心技术

| 组件 | 技术选型 | 理由 |
|------|----------|------|
| **语言** | Rust (Edition 2024) | 内存安全、零成本抽象、高性能 |
| **Web 框架** | Axum 0.8 | 异步、类型安全、Tower 生态、WebSocket 原生支持 |
| **Agent 框架** | Rig 0.7 | Rust 原生、支持 20+ LLM、10+ 向量库、流式输出 |
| **运行时** | Tokio | 成熟的异步运行时、丰富的生态 |
| **序列化** | serde + serde_json | 编译时类型检查、零成本反序列化 |

### 2.2 数据存储

| 存储 | 技术选型 | 用途 |
|------|----------|------|
| **关系数据库** | PostgreSQL 16+ | 用户数据、会话记录、持久化快照 |
| **缓存** | Redis 7+ | 会话状态缓存、消息队列、分布式锁 |
| **向量数据库** | Qdrant | 语义检索、RAG 上下文、知识关联 |
| **文件存储** | 本地 / S3 | 音频片段、导出文件 |

### 2.3 AI 服务

| 服务 | 技术选型 | 备注 |
|------|----------|------|
| **LLM** | OpenAI GPT-4-turbo | 支持 Claude、Gemini 作为备选 |
| **Embedding** | OpenAI text-embedding-3-small | 1536 维度 |
| **STT** | Whisper v3 | 支持本地部署 |
| **TTS** | ElevenLabs / Azure | 优先自然度 |

## 3. 架构原则

### 3.1 分层解耦

- **API 层**: 仅负责协议转换（HTTP/WebSocket → 内部消息）
- **核心层**: 业务逻辑、状态管理、Agent 编排
- **服务层**: 外部依赖抽象（STT/TTS、集成服务）
- **数据层**: 持久化、缓存、向量检索

### 3.2 依赖倒置

```rust
// 核心层定义 trait
pub trait VectorStore: Send + Sync {
    async fn search(&self, query: &str, limit: usize) -> Result<Vec<Document>>;
}

// 数据层实现
pub struct QdrantStore { ... }
impl VectorStore for QdrantStore { ... }
```

### 3.3 流式优先

- LLM 响应流式返回，降低首字延迟
- 音频数据流式处理，无需完整缓冲
- WebSocket 全双工通信

### 3.4 可观测性

- 结构化日志（tracing + tracing-subscriber）
- 分布式追踪（可选 opentelemetry）
- 指标导出（prometheus）

## 5. 性能目标

| 指标 | 目标 | 说明 |
|------|------|------|
| **首字延迟** | < 500ms | 用户说话到 Agent 开始响应 |
| **WebSocket 延迟** | < 100ms | 音频数据端到端 |
| **并发会话** | 1000+ | 单机支持 |
| **向量检索** | < 50ms | 语义搜索响应时间 |
| **内存占用** | < 200MB | 单会话 |

## 6. 安全考虑

### 6.1 认证授权

- JWT Token 认证
- Refresh Token 机制
- 会话隔离

### 6.2 数据安全

- 传输层 TLS 1.3
- 敏感数据加密存储
- API Key 环境变量隔离

### 6.3 速率限制

- WebSocket: 每会话 10 msg/s
- HTTP API: 每用户 100 req/min
- STT/TTS: 按第三方限制

---

## 7. 容错设计

### 7.1 降级策略

| 故障场景 | 降级方案 |
|----------|----------|
| LLM API 不可用 | 本地模型 (Ollama) |
| 向量库不可用 | 关键词检索 |
| Redis 不可用 | 内存缓存 |
| STT 不可用 | 文本输入模式 |

### 7.2 重试机制

- 指数退避重试
- 幂等性保证
- 断路器模式

---

## 8. 监控告警

### 8.1 关键指标

- 会话成功率
- 平均响应时间
- LLM Token 消耗
- 并发连接数
- 错误率

### 8.2 告警规则

- LLM API 错误率 > 5%
- 平均响应时间 > 2s
- WebSocket 断开率 > 10%
- 数据库连接池耗尽
